## Problem Statement

在企业微信群中，成员每天产生大量讨论；值班同学需要手工翻记录写「当日总结」，费时且易漏。目标是用**企微智能机器人**接入群聊，在**每天 18:00（工作日）**自动生成当日总结并**发回同一群**；另支持**手动触发**便于开发测试。

**部署约束（已澄清）：** 服务部署在**内网、无公网入站**；**允许出站**访问企微（`wss://openws.work.weixin.qq.com`）与 LLM API。「无公网」≠ 空气 gap。

## What Makes This Cool

- **零人工日常运行**：到点群里自动出现一条结构化日报，大家立刻能核对。  
- **内网友好**：用官方 **WebSocket 长连接**收消息，无需公网回调 URL。  
- **一键复现**：`手动触发` 让你在非 18:00 也能秒级验证整条 pipeline（学习/demo 的「哇」时刻）。

## Constraints

| 约束 | 说明 |
|------|------|
| 机器人类型 | **智能机器人（AI Bot）长连接**，非「仅 Webhook 发消息」群机器人（后者**不能读**群消息） |
| 连接 | `wss://openws.work.weixin.qq.com`；**单 Bot 同时仅 1 条有效连接**；心跳约 **30s** |
| 认证 | `BotID` + `Secret`（企业微信管理后台创建） |
| V1 范围 | **1 个群**；18:00 cron + **手动触发**；推回**同一群** |
| 部署 | 内网进程；出站可达企微与 LLM |
| 消息保留 | 默认 **7 天**（可配置）；仅用于日总结 |
| 消息类型 V1 | **仅文本**入库并参与总结；图片/文件仅记元数据（可选）或忽略 |
| 工作日 | 18:00 **周一至周五**（`Asia/Shanghai`）；法定节假日不单独处理（V1.1） |
| 技术栈（**已锁定**） | **Node.js + TypeScript + Bun** + `@wecom/aibot-node-sdk` + **bun:sqlite**（见 `doc/plan-wecom-daily-summary-eng.md`） |
| 部署实例 | **单进程单连接**；禁止多实例同时连同一 Bot（违反官方单连接限制） |

## Premises

1. 使用企微**智能机器人长连接**接收群消息（官方文档 path/101463）。  
2. 服务无公网入站，但可出站访问企微与 LLM。  
3. V1 单群；18:00 工作日自动总结并推群；提供手动触发。  
4. 日常零 PM 操作。  
5. 消息落库，默认保留 7 天。  
6. V1 总结使用**云端 LLM**（开发机可访问外网）；后续可换内网兼容端点。  
7. 需实现心跳、断线重连；遵守单连接限制。

## Approaches Considered

### Approach A: 单进程 SDK + SQLite（**已选**）

- **摘要：** 长连接收消息 → SQLite；定时/手动 → 读当日消息 → LLM 总结 → 长连接发群。  
- **工作量：** S | **风险：** 中  
- **优点：** 依赖少、最快闭环、适合学习  
- **缺点：** 多群扩展需重构  

### Approach B: 事件驱动 + 队列（BullMQ + Redis）

- **工作量：** M–L | **风险：** 低–中  
- **优点：** 幂等、重试、易扩展  
- **缺点：** V1 过重  

### Approach C: 官方「智能总结」+ 自建推送

- **风险：** 高（租户能力不确定）  
- **优点：** 总结质量可能更好  
- **缺点：** 学习路径不清晰  

## Recommended Approach

**Approach A**，技术要点如下。

### 架构（逻辑）

```
┌─────────────────┐     wss (出站)      ┌──────────────────┐
│  Bot 服务       │ ◄──────────────────►│ 企微 openws      │
│  (内网)         │                     │                  │
│  - WS 客户端    │                     └──────────────────┘
│  - 消息入库     │
│  - 定时/手动    │     HTTPS (出站)    ┌──────────────────┐
│  - LLM 总结     │ ───────────────────►│ LLM API          │
└────────┬────────┘                     └──────────────────┘
         │
         ▼
    ┌─────────┐
    │ SQLite  │  messages (chat_id, msg_id, sender, ts, body, ...)
    └─────────┘
```

### 模块划分

| 模块 | 职责 |
|------|------|
| `ws_client` | 建立长连接、`aibot_subscribe`、心跳、断线指数退避重连 |
| `message_handler` | 解析 `aibot_msg_callback`；去重（`msg_id`）；写入 DB |
| `store` | SQLite：`messages`、`summary_jobs`（幂等：`chat_id+date`） |
| `summarizer` | 按日筛选消息 → 构造 prompt → 调 LLM → 固定三段结构 |
| `publisher` | V1 **非流式单条文本**发回群内（对照 SDK 发送 API；超长按企微上限分段，建议单条 ≤4k 字） |
| `scheduler` | 18:00 cron；**V1 手动触发仅 CLI**：`python -m bot trigger-summary [--date YYYY-MM-DD]` |
| `retention` | 每日 02:00 删除超 `MSG_RETENTION_DAYS` 的消息（在当日总结完成后执行） |

### 总结模板（V1 固定）

1. **今日要点**（带时间 + 发送人，可回溯）  
2. **未决 / 待确认**  
3. **需关注**（@ 相关人、deadline、风险）  

> 防编造：prompt 要求仅基于当日 DB 中的消息；无消息则发「今日无有效群消息」。

### SQLite Schema（V1）

**`messages`**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 自增 |
| `msg_id` | TEXT UNIQUE | 企微消息去重键 |
| `chat_id` | TEXT | 与 `WECOM_CHAT_ID` 一致（回调里的会话/群标识，实施 Spike 0 时对照 SDK 字段名） |
| `sender_id` | TEXT | 发送者 userid |
| `sender_name` | TEXT | 展示名 |
| `msg_type` | TEXT | V1 固定 `text` |
| `body` | TEXT | 文本正文 |
| `raw_json` | TEXT | 原始回调（调试） |
| `created_at` | DATETIME | 入库时间 |

索引：`(chat_id, created_at)`。

**`summary_jobs`**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | |
| `chat_id` | TEXT | |
| `job_date` | TEXT | `YYYY-MM-DD` |
| `status` | TEXT | 见状态机 |
| `error` | TEXT | 失败原因 |
| `started_at` / `completed_at` | DATETIME | |

唯一约束：`UNIQUE(chat_id, job_date)`。

### 总结任务状态机

```
pending → summarizing → publishing → completed
                  ↘ failed（任阶段，error 字段记录）
```

- LLM 调用失败：最多重试 **2 次**，指数退避 1s/2s。  
- 推送失败：重试 **3 次**；仍失败则 `failed`，手动 CLI 可重跑同日。  
- **启动补跑**：进程启动时扫描 `job_date=today AND status NOT IN (completed)` → 自动继续。

### 定时与幂等

- Cron：`0 18 * * 1-5`（时区 `Asia/Shanghai`，库内显式绑定 `CRON_TZ`）。  
- 任务键：`{chat_id}:{YYYY-MM-DD}`；已 `completed` 则跳过（防重复推送）。  
- 失败：记录 `failed` + 原因；`python -m bot trigger-summary` 可重试同日。

### LLM 输入边界（V1）

| 规则 | 值 |
|------|-----|
| 单日最多纳入消息 | **500 条**（超出按时间均匀采样） |
| 单条正文上限 | **2000 字符**（超出截断） |
| 超时 | LLM 调用 **120s**；整体任务目标 **5 分钟内** 推送到群 |
| 防编造 | Prompt 仅允许引用提供的消息列表；验收时随机 3 条要点须能 `JOIN` 到 `messages` |

### 配置（环境变量）

| 变量 | 说明 |
|------|------|
| `WECOM_BOT_ID` | 机器人 ID |
| `WECOM_BOT_SECRET` | 密钥 |
| `WECOM_CHAT_ID` | V1 目标群 ID |
| `LLM_API_BASE` / `LLM_API_KEY` / `LLM_MODEL` | LLM 端点 |
| `CRON_TZ` | 默认 `Asia/Shanghai` |
| `MSG_RETENTION_DAYS` | 默认 `7` |

### 测试策略（无公网**入站**）

| 层级 | 做法 |
|------|------|
| 单元 | mock LLM 返回；mock 消息入库 |
| 集成 | 内网连真实 `openws`（出站）；或录制 JSON fixture 测 handler |
| 手动 | CLI `python -m bot trigger-summary`，不等到 18:00 |
| 注意 | 不要用「仅 Webhook 群机器人」测读消息——能力不匹配 |

### 官方替代方案（知悉即可）

企微 **4.1.38+** 群内「智能总结」可能满足部分需求；本设计以**自建**为主，便于学习长连接与定时 pipeline。若租户已开通且 API 稳定，V2 可评估 Approach C。

## Prerequisites（实施前检查）

1. 租户已开通 **智能机器人（AI Bot）** 能力（未开通则项目阻塞，无 Plan B 读群消息）。  
2. **Spike 0（≤0.5 天）**：连 `openws` → 收一条文本 → 经长连接发一条文本回群 → 从回调日志确认 `chat_id` 字段名 → 写入 `WECOM_CHAT_ID`。  
3. 出站白名单：`openws.work.weixin.qq.com` + 所选 LLM 域名。

## Open Questions

1. **LLM 选型**：OpenAI 兼容 / DeepSeek / 其他？（V1 默认任选其一写死在 `.env`；编码前需定）  

## Success Criteria

- [ ] 长连接稳定运行 ≥24h：断连后 **2 分钟内**恢复；24h 内可恢复断连 **≤5 次**  
- [ ] 群消息入库率：文本消息不丢（以 `msg_id` 去重统计）  
- [ ] 18:00 或手动触发后 **5 分钟内** 群内出现当日总结  
- [ ] 总结中每条要点可对应到 DB 中的原始消息（时间+发送人）  
- [ ] 内网部署、无公网 IP，仅出站即可完成功能验证

## Security & Ops（V1 最小）

- Secret 仅来自环境变量 / `.env`，**不入库**；`.env` 权限 `600`。  
- SQLite 文件仅本机可读；禁止导出到公网。  
- 结构化日志：收消息、任务状态迁移、LLM 耗时、推送结果。  
- **稳态零人工**：一次性完成后台安装与配置后，运行期无需 PM 操作（bootstrap 除外）。
