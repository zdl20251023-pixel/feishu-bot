### Schema 修订（相对设计文档）

**messages** 增加：

| 字段 | 类型 | 说明 |
|------|------|------|
| `msg_ts` | INTEGER | 企微消息 Unix 秒（当日筛选依据） |

索引：`(chat_id, msg_ts)`。

**summary_jobs** 增加：

| 字段 | 类型 | 说明 |
|------|------|------|
| `wecom_msg_id` | TEXT NULL | 已发出总结的消息 ID，重试前检查 |

状态：`pending | summarizing | publishing | completed | failed`

### 当日边界

- 时区：`Asia/Shanghai`
- 当日范围：`[当天 00:00:00, 当天 23:59:59]` 按 `msg_ts` 过滤
- Cron：`0 18 * * 1-5`；CLI：`bun run src/cli.ts trigger-summary [--date=YYYY-MM-DD]`

### 后台 Job（D2 落地）

- WS `onMessage`：仅 `INSERT` + 去重，**不**调用 LLM
- `summaryJobRunner.run(date)`：由 cron / CLI 触发，`async`，内部 `fetch` LLM
- 若 LLM >30s，事件循环仍可处理 WS（Node fetch 非阻塞；切忌同步 CPU 密集）

## NOT in scope（V1 明确不做）

| 项 | 理由 |
|----|------|
| 多群 | 设计 V1 单群 |
| 图片/文件内容理解 | 仅文本；附件最多记元数据 |
| Redis/BullMQ 队列 | Approach A；单进程足够 |
| 法定节假日日历 | V1.1 |
| HTTP admin API | 仅 CLI 手动触发 |
| 第二进程/双连接 | 违反企微单连接 |

## Failure modes（关键）

| 场景 | 测试 | 错误处理 | 用户可见 |
|------|------|----------|----------|
| openws 断连 | E2E/集成 | 指数退避重连 | 短暂收不到消息 |
| LLM 超时 120s | 单测 mock | → failed，可 CLI 重跑 | 当天无总结 |
| 推送假失败双发 | 单测 wecom_msg_id | 3B 方案 | 两条总结（已防） |
| 空消息日 | 单测 + eval | 固定「今日无有效群消息」 | 一条说明 |
| 进程 18:00 重启 | 启动补跑单测 | catch-up job | 可能延迟几分钟 |
