## Problem Statement

在飞书群协作中，成员每天产生大量讨论，但信息容易散落在聊天记录里：谁说了什么、今天推进了什么、哪些事项还没完成，往往需要人工翻记录。目标是做一个**飞书智能机器人常驻服务**，同时支持：

1. **即时对话**：私聊和群聊 @ 机器人时，有 LLM 则智能回复，无 LLM 则模板回复。  
2. **群聊记忆**：未 @ 机器人的群消息也要入库，供后续分析。  
3. **定时总结**：工作日自动发群聊阶段性总结和下班总结。  
4. **任务跟进**：从群消息中识别“安排任务”类内容，定时在原群提醒未完成任务。

部署与学习约束：服务在内网可出站访问飞书 API 与 LLM；凭证由 `lark-cli` 管理；V1 基于 **Bun + TypeScript + `lark-cli`**，不修改 `src/`（企微）代码。

## What Makes This Cool

- **一个机器人，三种能力**：既能当群助手聊天，又能当“群秘书”做日总结，还能当“任务催办员”。  
- **你已打通最难的一段**：私聊收发、群 @ 收事件、`lark-cli` 长连接常驻。  
- **可演示性强**：工作日 10:00 催任务、12:00 午报、18:00 晚报，群里会有明显“有人在帮你记事情”的感觉。

## Constraints

| 约束 | 说明 |
|------|------|
| 技术栈 | Bun + TypeScript；事件与发消息经 `lark-cli`；**不修改 `src/`** |
| 事件源 | `lark-cli event consume im.message.receive_v1 --as bot` |
| 数据源前提（V1） | **假设**事件流能收到机器人可见的**全部群消息**（含未 @）；实现前需实测验证 |
| 监控范围 | **所有机器人可见的群** + 私聊；不做单群白名单 |
| 消息保留 | **30 天**；超期清理 |
| 消息类型 V1 | **仅 text** 入库并参与 LLM；其他类型可记元数据或忽略 |
| 时区 | `Asia/Shanghai`（与现有 `.env` 中 `CRON_TZ` 一致） |
| LLM | OpenAI 兼容 API；无 `LLM_API_KEY` 时走规则/模板 |
| 身份 | Bot 身份（`--as bot`）；用户身份未登录不影响 Bot 收消息 |

## Premises

1. V1 使用飞书**智能机器人** + `lark-cli` 事件消费，不引入企微 SDK。  
2. V1 **假设** `im.message.receive_v1` 能推送机器人可见范围内的群聊消息（含未 @）；若实测仅 @ 才推送，则“未 @ 全量存储”需降级或改会话存档方案。  
3. 监控范围为机器人可见的**全部群**与私聊，保留 30 天消息。  
4. 定时总结：工作日 **12:00**、**18:00** 各发一次群总结，并支持手动触发。  
5. 任务识别采用**规则筛选 + LLM 抽取**；任务完成仅接受**明确指令**（如 `完成 T-001`）。  
6. 任务提醒发回**原群**，工作日 **10:00** 单独提醒未完成任务，总结中再次汇总。  
7. 私聊与群 @：**普通文本走 LLM**；`总结` / `任务` / `帮助` / `完成` 等走专门命令处理。

## Platform Boundary（必读）

| 场景 | V1 设计假设 | 若实测不符 |
|------|-------------|-----------|
| 私聊文本 | 能收、能回 | 查 Bot 权限与事件订阅 |
| 群聊 @ 机器人 | 能收、能回 | 同左 |
| 群聊未 @ | **能收并入库** | 降级为仅存储已推送消息，或立项「会话存档」第二数据源 |
| 发送总结/提醒 | `lark-cli im +messages-send` 或 `+messages-reply` | 查发消息权限 |

**Office Hours 结论：** 未 @ 全量存储是整份需求的数据基础；必须在编码前用一条**不 @ 的普通群消息**完成实测并记录结果。

## Functional Requirements（已确认）

### 1. 私聊

| 条件 | 行为 |
|------|------|
| 用户发普通文本 | 有 LLM → 智能回复；无 LLM → 固定模板说明 |
| 用户发命令词 | 见「命令词」表，不走通用闲聊 LLM |
| 机器人自己发的消息 | 忽略（防自回环，如内容以 `🤖` 开头） |

### 2. 群聊 — @ 机器人

| 条件 | 行为 |
|------|------|
| 用户 @ 机器人发普通文本 | 同私聊：LLM 或模板回复（回复原消息） |
| 用户 @ 机器人发命令词 | 执行对应功能（总结、任务查询等） |
| 仅 @ 无正文 | 可回复帮助说明（V1 可选） |

### 3. 群聊 — 未 @ 机器人

| 条件 | 行为 |
|------|------|
| 任意可见群文本消息 | **写入 SQLite**，不参与即时回复（静默采集） |
| 图片/文件等非文本 | V1 不解析正文；可选记 `message_type` + 占位 |

### 4. 定时总结

| 触发 | 行为 |
|------|------|
| 工作日 12:00 | 对每个有消息的群，生成**阶段性总结**并发送到该群 |
| 工作日 18:00 | 对每个有消息的群，生成**下班总结**并发送到该群 |
| 手动 | 用户发 `总结` 或 CLI/内部命令，立即生成当日（或指定范围）总结 |
| 无消息日 | 可跳过或发“今日无有效群消息” |

总结内容建议结构（Markdown）：

- 今日要点  
- 未决 / 待确认  
- 需关注  
- 未完成任务摘要（与 10:00 提醒呼应）

### 5. 任务识别与跟进

**识别（混合）：**

1. 规则候选：消息含 `安排`、`负责`、`截止`、`TODO`、`跟进`、`完成` 等，或 @某人 + 时间表达。  
2. LLM 抽取字段：任务描述、负责人（open_id/姓名若可解析）、截止时间、来源 `chat_id`、来源 `message_id`、置信度。

**提醒：**

| 时间 | 行为 |
|------|------|
| 工作日 10:00 | 在原群发送「未完成任务」提醒列表 |
| 工作日 12:00 / 18:00 | 在总结中附带未完成任务小节 |

**完成：**

- 仅接受明确指令，例如：`完成 T-001`、`已处理 T-001`、`done T-001`  
- V1 **不做** LLM 自动判断完成

### 6. 命令词（私聊与群 @）

| 命令/关键词 | 行为 |
|-------------|------|
| `总结` / `summary` | 生成当前会话或当前群当日总结 |
| `任务` / `任务状态` | 列出未完成任务 |
| `帮助` / `help` | 返回可用命令说明 |
| `完成 T-xxx` | 将任务标记为已完成 |
| 其他文本 | LLM 智能回复（无 LLM 则模板） |

## Approaches Considered

### Approach A: 单进程 `src2` 渐进扩展（**推荐**）

- **摘要：** `event consume` 收事件 → 写 SQLite → 路由（私聊回复 / 群 @ 回复 / 静默入库）→ cron 跑总结与任务提醒。  
- **工作量：** M | **风险：** 中（未 @ 全量依赖平台实测）  
- **优点：** 延续现有 `src2` 联调成果；模块可参考 `src/` 的 store/scheduler/summarizer 模式  
- **缺点：** `index.ts` 需拆分为多模块  

### Approach B: 事件进程 + 调度进程拆分

- **摘要：** 进程 1 只消费事件写库；进程 2 负责 cron 与 LLM 作业。  
- **工作量：** L | **风险：** 低–中  
- **优点：** 故障隔离、易扩展  
- **缺点：** 学习/demo 阶段运维偏重  

### Approach C: 未 @ 全量走会话内容存档

- **摘要：** 智能机器人负责对话；全群历史用飞书会话存档 API 拉取。  
- **工作量：** XL | **风险：** 高  
- **优点：** 若事件流不全，这是唯一“真·全群”方案  
- **缺点：** 与当前 `lark-cli` 路线割裂，合规与开发成本高  

## Recommended Approach

**Approach A**：在 `src2/` 下拆模块一次建齐，参考企微项目目录思想但不复制企微 API。

### 目标目录（建议）

```text
src2/
  index.ts              # 启动：事件消费 + cron 注册
  config.ts             # 环境变量
  event-consumer.ts   # lark-cli 子进程与 NDJSON 解析
  message-router.ts   # 私聊 / 群@ / 群静默 分流
  store.ts              # SQLite：messages, tasks, summary_runs
  reply.ts              # lark-cli im 发消息/回复
  summarizer.ts         # LLM / 模板总结
  task-extractor.ts     # 规则 + LLM 抽任务
  task-reminder.ts      # 10:00 提醒
  scheduler.ts          # 12:00 / 18:00 cron
  commands.ts           # 总结/任务/帮助/完成
  logger.ts
  types.ts
```

### 逻辑架构

```text
                    lark-cli event consume
                              │
                              ▼
                      ┌───────────────┐
                      │ message-router │
                      └───────┬───────┘
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
    [p2p 即时回复]    [group @ 即时回复]   [group 静默入库]
          │                 │                 │
          └─────────────────┴────────┬────────┘
                                     ▼
                              SQLite (30d)
                                     │
              ┌──────────────────────┼──────────────────────┐
              ▼                      ▼                      ▼
        task-extractor          summarizer            task-reminder
              │                      │                      │
              └──────────────────────┴──────────────────────┘
                                     │
                              cron 10:00 / 12:00 / 18:00
                                     │
                              lark-cli 发群消息
```

## Success Criteria

| # | 标准 |
|---|------|
| 1 | 私聊发任意文本，机器人能回复（LLM 或模板） |
| 2 | 群内 @ 机器人，能智能回复或执行命令 |
| 3 | 群内未 @ 发普通文本，日志/DB 中有记录（**前提：平台推送**） |
| 4 | 工作日 12:00、18:00 目标群收到总结（有消息时） |
| 5 | 工作日 10:00 目标群收到未完成任务提醒（有 open 任务时） |
| 6 | `完成 T-001` 后，该任务不再出现在下次提醒中 |
| 7 | 进程重启后，30 天内历史消息仍可被总结（SQLite 持久化） |

## Distribution Plan

- **运行：** `bun run src2/index.ts`（或拆 CLI 后 `bun run start:lark`）  
- **配置：** 项目根 `.env` + `lark-cli` 已 `auth`（Bot ready）  
- **数据：** `./data/lark-bot.db`（建议与企微 `bot.db` 分离）  
- **部署：** 内网单机常驻；单进程；需保证 `event consume` 子进程 stdin 不断开
